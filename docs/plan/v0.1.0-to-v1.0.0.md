## 1. Visão do Produto

**Visionary** é uma plataforma SaaS de gestão financeira integrada que conecta parceiros de negócio. Cada empresa tem seu próprio tenant isolado com:

- **Painel financeiro** — dashboard com métricas em tempo real
- **Conciliação bancária** — upload de extratos e matching automático
- **Fluxo de caixa** — entradas, saídas, projeções
- **Conexão entre parceiros** — empresas podem se conectar para compartilhar transações, faturas e reconciliações
- **Multi-tenant real** — isolamento total por empresa, com backoffice centralizado

## 3. Domínio — Modelos de Dados

### 3.1 Novos modelos (Schema Prisma)

```prisma
// ─── Núcleo Financeiro ───

model BankAccount {
  id            String   @id @default(cuid())
  tenantId      String   @map("tenant_id")
  name          String                       // "Itaú PJ", "Nubank"
  bank          String                       // código do banco
  agency        String?
  accountNumber String   @map("account_number")
  balance       Decimal  @default(0)
  currency      String   @default("BRL")
  isActive      Boolean  @default(true) @map("is_active")
  createdAt     DateTime @default(now()) @map("created_at")
  updatedAt     DateTime @updatedAt @map("updated_at")

  tenant        Tenant      @relation(fields: [tenantId], references: [id])
  transactions  Transaction[]
  bankStatements BankStatement[]

  @@unique([tenantId, name])
  @@map("bank_accounts")
}

model Transaction {
  id              String    @id @default(cuid())
  tenantId        String    @map("tenant_id")
  bankAccountId   String    @map("bank_account_id")
  type            TransactionType  // INCOME, EXPENSE, TRANSFER
  amount          Decimal
  description     String
  categoryId      String?   @map("category_id")
  date            DateTime
  dueDate         DateTime? @map("due_date")
  status          TransactionStatus @default(PENDING)
  partnerId       String?   @map("partner_id")      // transação vinculada a parceiro
  reconciledAt    DateTime? @map("reconciled_at")
  metadata        Json?                              // dados extras flexíveis
  createdAt       DateTime  @default(now()) @map("created_at")
  updatedAt       DateTime  @updatedAt @map("updated_at")

  tenant          Tenant          @relation(fields: [tenantId], references: [id])
  bankAccount     BankAccount     @relation(fields: [bankAccountId], references: [id])
  category        Category?       @relation(fields: [categoryId], references: [id])
  partner         Partner?        @relation(fields: [partnerId], references: [id])
  attachments     TransactionAttachment[]
  reconciliations Reconciliation[]

  @@index([tenantId, date])
  @@index([bankAccountId])
  @@map("transactions")
}

model Category {
  id        String @id @default(cuid())
  tenantId  String @map("tenant_id")
  name      String
  type      TransactionType
  color     String?                              // hex para UI
  icon      String?                              // nome do ícone (Lucide)
  parentId  String? @map("parent_id")            // categoria pai (hierarquia)

  tenant    Tenant      @relation(fields: [tenantId], references: [id])
  parent    Category?   @relation("CategoryHierarchy", fields: [parentId], references: [id])
  children  Category[]  @relation("CategoryHierarchy")
  transactions Transaction[]

  @@unique([tenantId, name, type])
  @@map("categories")
}

model BankStatement {
  id             String   @id @default(cuid())
  tenantId       String   @map("tenant_id")
  bankAccountId  String   @map("bank_account_id")
  fileName       String   @map("file_name")
  periodStart    DateTime @map("period_start")
  periodEnd      DateTime @map("period_end")
  status         StatementStatus @default(PENDING)  // UPLOADED, PROCESSING, RECONCILED, ERROR
  totalEntries   Int      @default(0) @map("total_entries")
  totalMatched   Int      @default(0) @map("total_matched")
  uploadedAt     DateTime @default(now()) @map("uploaded_at")

  tenant         Tenant       @relation(fields: [tenantId], references: [id])
  bankAccount    BankAccount  @relation(fields: [bankAccountId], references: [id])
  reconciliations Reconciliation[]

  @@map("bank_statements")
}

model Reconciliation {
  id              String   @id @default(cuid())
  tenantId        String   @map("tenant_id")
  transactionId   String   @map("transaction_id")
  statementId     String   @map("statement_id")
  matchedAmount   Decimal  @map("matched_amount")
  confidence      Float                          // 0.0 a 1.0 — score do match
  status          String   @default("MATCHED")   // MATCHED, UNMATCHED, CONFLICT
  createdAt       DateTime @default(now()) @map("created_at")

  transaction     Transaction    @relation(fields: [transactionId], references: [id])
  statement       BankStatement  @relation(fields: [statementId], references: [id])

  @@unique([transactionId, statementId])
  @@map("reconciliations")
}
```

### 3.2 Modelos de Conexão entre Parceiros

```prisma
model Partner {
  id              String    @id @default(cuid())
  tenantId        String    @map("tenant_id")     // quem criou a conexão
  connectedTenantId String  @map("connected_tenant_id")  // tenant do parceiro
  status          PartnerStatus @default(PENDING)  // PENDING, ACTIVE, BLOCKED
  nickname        String?                          // apelido local
  connectionRequestedAt DateTime @default(now()) @map("connection_requested_at")
  connectionAcceptedAt DateTime? @map("connection_accepted_at")

  tenant          Tenant         @relation("PartnerTenant", fields: [tenantId], references: [id])
  connectedTenant Tenant         @relation("ConnectedTenant", fields: [connectedTenantId], references: [id])
  invoices        PartnerInvoice[]
  transactions    Transaction[]

  @@unique([tenantId, connectedTenantId])
  @@map("partners")
}

model PartnerInvoice {
  id          String   @id @default(cuid())
  partnerId   String   @map("partner_id")
  sentById    String   @map("sent_by_id")          // tenant que enviou
  number      String                              // número da fatura
  description String?
  amount      Decimal
  dueDate     DateTime @map("due_date")
  status      InvoiceStatus @default(PENDING)      // PENDING, ACCEPTED, DISPUTED, PAID
  sentAt      DateTime @default(now()) @map("sent_at")
  paidAt      DateTime? @map("paid_at")

  partner     Partner @relation(fields: [partnerId], references: [id])
  sentBy      Tenant  @relation(fields: [sentById], references: [id])

  @@map("partner_invoices")
}
```

## 4. API — Routers tRPC

### 4.1 Manter (existente)
- `auth` — autenticação (register, login, session, changePassword)
- `tenant` — gestão de tenants (list, bySlug, create, update)
- `team` — convites e gestão de membros
- `user` — perfil do usuário

### 4.2 Novos routers

| Router | Procedures | Descrição |
|--------|-----------|-----------|
| `transaction` | `list`, `byId`, `create`, `update`, `delete`, `bulkCreate`, `summary` | CRUD de transações + resumo (entradas/saídas/saldo) |
| `category` | `list`, `create`, `update`, `delete` | Categorias hierárquicas de receita/despesa |
| `bankAccount` | `list`, `create`, `update` | Contas bancárias |
| `statement` | `list`, `upload`, `process`, `summary` | Upload e processamento de extratos (OFX/CSV) |
| `reconciliation` | `list`, `match`, `unmatch`, `bulkMatch`, `stats` | Conciliação manual e automática |
| `partner` | `list`, `requestConnection`, `acceptConnection`, `block`, `listPending` | Gestão de parceiros |
| `invoice` | `list`, `send`, `accept`, `dispute`, `pay`, `listByPartner` | Faturas entre parceiros |
| `dashboard` | `metrics`, `cashflow`, `categoryBreakdown`, `pendingInvoices` | Métricas agregadas para o dashboard |

### 4.3 Regras de negócio importantes

- **Transações**: sempre escopo de tenant. Criação valida `bankAccountId` pertence ao tenant.
- **Conciliação**: matching automático por valor + data (±2 dias) + descrição similar. Confiança < 0.8 vai para revisão manual.
- **Parceiros**: conexão bidirecional. Ambos precisam aceitar. Só tenants ativos podem se conectar.
- **Faturas**: só podem ser enviadas para parceiros com status ACTIVE. Disputa congela o prazo.

## 10. Riscos e Trade-offs

| Risco | Mitigação |
|-------|-----------|
| Parser OFX instável | Testar com extratos reais de 3 bancos diferentes antes de liberar |
| Algoritmo de matching com falsos positivos | Threshold de confiança configurável; matches < 0.8 vão para revisão manual |
| Performance com muitas transações | Índices no Prisma; paginação server-side; agregações via raw SQL |
| Complexidade da UI multi-tenant + parceiros | Manter escopo enxuto na v1; parceiros só compartilham faturas (não transações individuais) |
| Next.js 15 → 16 breaking changes | Permanecer no 15 até estabilizar; migrar depois |
